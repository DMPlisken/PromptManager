JavaScript Errors Detected

Console errors caught during test execution and resolved

ERROR 1 TypeError: Cannot read properties of undefined (reading 'substring')
Thrown on the Sessions page when rendering session list items. The session_id field was undefined for sessions created before the UUID migration. The component called .substring(0, 8) without a null guard.
✓ Fixed by adding optional chaining: session_id?.substring(0, 8) ?? 'N/A'
ERROR 2 TypeError: Cannot read properties of undefined (reading 'machine_uuid')
MachineCard component crashed when the machine object from the API response was missing the machine_uuid field. This occurred for machines registered before the multi-machine feature was added.
✓ Fixed by adding fallback: machine?.machine_uuid ?? machine?.id ?? 'unknown'

Bugs Found and Fixed 4/4 RESOLVED

Identified during E2E test execution and manual review

CRITICAL Sessions page completely blank (React crash)
Affected: src/pages/SessionsPage.tsx · src/components/SessionListItem.tsx
The entire Sessions page rendered as a blank white screen. React's error boundary caught an unhandled TypeError: Cannot read properties of undefined (reading 'substring'). The root cause was SessionListItem calling session.session_id.substring(0, 8) to display a truncated ID, but some legacy session records returned null for session_id from the API. A single null session in the list crashed the entire page due to React's error propagation.
✓
Fix: Added optional chaining and fallback display value. Also added a top-level ErrorBoundary around the session list so a single bad record cannot take down the whole page.
src/components/SessionListItem.tsx · src/pages/SessionsPage.tsx
MEDIUM Test output showing "(no output received)" despite 5 messages
Affected: src/components/SessionOutput.tsx · src/hooks/useSessionStream.ts
The session output panel displayed "(no output received)" even when the WebSocket had delivered 5 messages. The issue was a race condition: the output component checked the message array length on its initial render, before the WebSocket hook had pushed messages into state. The dependency array on the useEffect that set the "no output" flag did not include the message array.
✓
Fix: Added messages.length to the useEffect dependency array. Changed the empty-state check to only trigger after a 2-second debounce, giving the stream time to connect.
src/components/SessionOutput.tsx
LOW Setup wizard creating stale pairing records on cancel
Affected: src/components/SetupWizard.tsx · src/api/machines.ts
When a user started the machine setup wizard and clicked "Cancel" at step 2 (after the pairing code was already generated), the backend retained the pending pairing record. Over time this accumulated orphaned pairing codes that could never be claimed, cluttering the machines table with status: 'pending_pairing' rows.
✓
Fix: Added a cleanup API call (DELETE /api/machines/pairing/{code}) in the wizard's cancel handler and useEffect cleanup function.
src/components/SetupWizard.tsx
MEDIUM MachineCard crash on undefined machine_uuid
Affected: src/components/MachineCard.tsx
The MachineCard component used machine.machine_uuid as a React key and in the status display. Machines registered before the multi-machine feature was added did not have this field, causing the component to throw when rendering the machine dashboard.
✓
Fix: Added a fallback chain: machine?.machine_uuid ?? machine?.id ?? 'unknown'. Also added a TypeScript type guard in the machine list mapper.
src/components/MachineCard.tsx
