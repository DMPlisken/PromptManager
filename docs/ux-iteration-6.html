UX Iteration 6 — Backend: Stale Pairing Cleanup

Focus: Auto-cleanup of expired pairing records, database hygiene on GET /machines

Theme

Iteration 6 shifted focus to backend polish. The UX expert review in iteration 5 identified that stale pairing records accumulate in the database when users generate pairing codes but never complete the pairing. This iteration adds automatic cleanup on every GET /machines request.

Changes Made

Stale Pairing Record Cleanup

Added automatic cleanup logic at the top of the GET /api/machines endpoint.

machines.py
  • On every list_machines call, queries for Machine records with status == "pairing" and pairing_expires_at older than 1 hour
  • Deletes all matching stale records in a single pass
  • Logs each deleted record with its machine_id, pairing_code, and expiry time
  • Only commits if stale records were found (no unnecessary DB writes)
  • This is a lightweight check — the WHERE clause uses indexed columns
stale_cutoff = datetime.now(timezone.utc) - timedelta(hours=1) stale_query = select(Machine).where( Machine.status == "pairing", Machine.pairing_expires_at < stale_cutoff, ) stale_result = await db.execute(stale_query) stale_machines = stale_result.scalars().all() for stale in stale_machines: logger.info("cleanup_stale_pairing", machine_id=stale.id, ...) await db.delete(stale) if stale_machines: await db.commit()

Why This Approach

  • Piggybacks on existing traffic — no need for a separate background job or cron task
  • Bounded work — stale pairings are rare (at most a few per day), so the cleanup is near-instantaneous
  • BUG-003 complement — the frontend already cleans up on wizard cancel (BUG-003 fix), but this catches edge cases like browser crashes or network failures during the wizard
  • 1-hour grace period — pairing codes expire after 15 minutes, but we wait a full hour before deleting to avoid race conditions with slow connections
